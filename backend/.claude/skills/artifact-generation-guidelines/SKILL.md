# Artifact Generation Guidelines

## 目标

这个 skill 用来约束客户需求转译台里三类交付物的触发时机和生成边界：

- `document`
- `page_solution`
- `interaction_flow`

它解决两个问题：

1. 什么时候应该把当前分析结果整理成交付物
2. 三类交付物各自应该长成什么样，避免内容跑偏

## 触发原则

只有当上下文已经形成一版可被整理的分析结果，并且用户语义上是在要求“把它做出来 / 整理出来 / 给我看一版”时，才触发交付物生成。

判断时看三层信号：

1. 当前用户消息
2. 最近几轮对话上下文
3. 当前项目沉淀是否已经足够支撑草稿

不要把“是否包含某几个关键词”当成唯一标准。

下面这些表达通常都可以视为要求生成交付物：

- “那就做啊”
- “来一版我看看”
- “按这个整理出来”
- “那你直接出稿吧”
- “先给我一版页面方案”
- “把这部分沉淀成文档”

下面这些情况一般不要触发：

- 用户还在补充事实，没有要求整理结果
- 用户只是问“文档里写了什么”或“页面上现在有什么”
- 当前证据和沉淀还过于空，连草稿边界都说不清

## 类型判断

### `document`

适合这些场景：

- 用户要需求稿、方案稿、需求摘要、分析文档
- 用户要把当前结论整理成可继续推进的书面材料

输出重点：

- 项目目标
- 真实需求
- 范围边界
- 冲突与风险
- MVP 能力包
- 验收建议

### `page_solution`

适合这些场景：

- 用户要页面方案、页面结构、信息架构、页面原型方向
- 用户要看未来系统应该有哪些页面、各页面承担什么职责

输出重点：

- 页面分工
- 信息结构
- 主要模块
- 页面之间关系

硬约束：

- 这类产物固定代表“页面方案 HTML 原型”，不是纯文字页面说明
- 标题用自然中文，能反映页面场景或页面任务，不要写成泛化的“设计稿”
- 内容必须可预览，且主要表达页面结构、模块分区和信息组织

不要把它写成交互步骤文档。

### `interaction_flow`

适合这些场景：

- 用户要交互稿、交互流程、关键操作链路
- 用户要看用户如何一步步完成任务

输出重点：

- 关键步骤
- 每步动作
- 状态变化
- 页面衔接
- 异常分支

硬约束：

- 这类产物固定代表“交互稿 HTML 原型”，不是说明文档的 HTML 包装版
- 标题用自然中文，能反映当前交互场景或任务，不要写成“流程说明”“说明文档”
- 必须让用户看见可操作的界面结构、关键按钮或入口、状态反馈和流程推进
- 内容重点是流程推进、动作反馈、分支处理，不是页面清单，也不是大段文字说明

不要把它写成单纯的页面清单。

## 生成边界

1. 可以输出草稿，但必须明确哪些内容是“待确认”。
2. 不要为了凑完整而编造业务事实。
3. 不要把内部状态桶、工具过程、文件路径、方法论术语直接暴露给用户。
4. 如果用户没有指定类型，优先选最贴近当前诉求的一种，不要一次性生成三种。
5. 如果用户明确要求多种交付物，可以同时请求多种类型。

## 与聊天回复的关系

当本轮决定触发交付物时，面向用户的自然回复应该：

1. 先说清楚你正在整理什么
2. 简要说明会基于哪些已确认理解和边界来整理
3. 不要描述内部工具、文件写入、目录操作
4. 不要输出 `<think>`、`tool_call`、`TodoWrite`
