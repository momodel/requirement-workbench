# 客户需求转译台 PRD（最终版）

## 1. 产品概述

- 产品名称：客户需求转译台 / Requirement Workbench
- 产品定位：面向 AI 项目前期的需求分析智能体工作台
- 一句话定义：把“客户说得很乱的需求”转成“产品、设计、研发、售前都能继续推进的执行稿”
- 方法原则：分析优先，生成其次；AI 起草，人确认排版

本产品不是一个泛化聊天机器人，也不是纯文档总结工具，而是一个带阶段推进、证据追溯、需求收敛与方案定义能力的工作台。它接住聊天、文档、图片等原始输入，在多轮澄清过程中持续沉淀结构化结果，最终输出可执行需求、MVP 方案和原型建议。

## 2. 要解决的问题

在 AI 项目前期，客户通常给出的不是执行需求，而是模糊愿望、问题抱怨或方案倾向。实际团队需要花大量时间完成以下工作：

- 收客户资料，整理上下文
- 理解业务流程、角色和系统边界
- 区分事实、假设、目标、边界与风险
- 判断哪些问题适合 AI，哪些不适合
- 输出第一版 MVP 方案和原型方向

当前痛点主要体现在：

- 输入碎片化：聊天、文档、图片、会议纪要分散
- 理解不稳定：每轮沟通都可能丢背景
- 分析质量不稳定：高度依赖个人经验
- 方案与设计脱节：需求分析难以直接交给设计和研发
- 结论不可追溯：后续很难定位某条判断来自哪份资料

## 3. 产品定位与边界

### 3.1 产品定位

客户需求转译台是一个“需求分析 + 方案收敛 + 设计交付”工作台。它面向咨询顾问、售前顾问、产品经理、解决方案架构师，也适合企业内部数字化负责人用于项目预研。

### 3.2 不做什么

- 不做通用聊天机器人
- 不做单纯的知识库问答
- 不替代资深顾问做最终承诺
- 不在 v1 自动生成高保真视觉设计
- 不在 v1 直接输出代码级任务拆解

### 3.3 核心价值

- 对客户侧：把模糊诉求讲清楚，明确 AI 的可做与不可做
- 对交付侧：把需求分析结果直接变成可推进的方案和原型底稿
- 对团队侧：沉淀统一的方法论、阶段产物和决策证据

## 4. 目标用户与使用场景

### 4.1 直接用户

- AI 转型咨询顾问
- 售前顾问
- 产品经理
- 解决方案架构师
- 企业内部数字化负责人

### 4.2 核心场景

- 售前首次接单：快速判断是否值得继续推进
- 咨询需求澄清：从资料和访谈里还原真实业务
- 方案定义：输出 AI 适配判断和 MVP 草案
- 设计交付：把方案转成页面结构、交互流和原型建议

## 5. 产品工作流（五阶段）

### 5.1 需求接入

目标：接住客户原话和初始资料，建立项目卡片与缺失清单。

输出：项目卡片、资料清单、初步问题定义、缺失资料清单。

### 5.2 业务理解

目标：还原角色、流程、术语、系统与约束。

输出：项目画像、角色清单、现状流程、痛点与约束。

### 5.3 需求收敛

目标：把事实、假设、目标、边界拆开，只围绕关键缺口追问。

输出：已确认事实、待确认项、范围边界、真实需求定义、优先级排序。

### 5.4 方案定义

目标：判断 AI 适配度，定义 MVP、风险边界和实施前提。

输出：AI 适配判断、MVP 草案、能力包、风险边界、验收指标建议。

### 5.5 设计交付

目标：把方案翻译成页面结构、交互流和可交付原型。

输出：页面清单、信息架构、关键交互流、原型建议、设计交付稿。

## 6. 界面与交互结构

采用“顶部状态栏 + 左中右三栏工作台”结构：

- 顶部：项目名、阶段、资料数、待确认项、AI 适配度、版本状态
- 左栏：资料与证据，包括项目概况、资料列表、流程文件、术语、页面资产
- 中栏：聊天与澄清，包括当前阶段、关键追问、当前理解、快捷动作
- 右栏：项目收敛结果，包括已确认事实、范围边界、待确认项、MVP 草案、指标卡片

交互原则：

- 不让聊天吞掉结论，所有关键结果都沉淀到右栏
- 不无限追问，每轮只问最关键的 3-5 个问题
- 不先给大方案，先补缺口、再收敛、最后起草
- 每条重要结论都尽量能回溯证据来源

## 7. 需求分析智能体能力定义

### 7.1 输入

- 聊天记录
- 文档（PDF / Word / Markdown / Excel）
- 图片（截图、扫描件、流程图）

### 7.2 AI 负责的事

- 信息抽取：背景、目标、流程、角色、约束、风险、痛点
- 冲突识别：识别版本冲突、口径冲突、系统边界冲突
- 缺口识别：识别缺失资料、关键未确认项
- 需求收敛：从模糊表达中形成真实需求定义
- 方案起草：输出 MVP 方案和原型建议

### 7.3 AI 不直接负责的事

- 最终需求确认
- 最终优先级拍板
- 高责任风险承诺
- 正式对外交付版排版与发布

### 7.4 人机协作原则

- AI 负责分析和起草
- 人负责确认、修改、取舍和最终排版

### 7.5 方法论应用机制（平台内生）

本产品的方法论不是给团队背诵的概念集合，而是内化为智能体平台的处理机制。平台不直接“套用某个框架”，而是把不同方法论拆成不同的内部职责，共同服务于一个目标：把客户原话和零散资料，收敛成可确认、可追溯、可继续推进的需求定义。

#### 7.5.1 方法论与平台模块的对应关系

- `BABOK`：作为需求抽取骨架，保证平台在分析时不遗漏业务目标、角色、流程、系统、规则、约束、风险、成功标准等关键维度。
- `JTBD`：作为真实需求判断机制，用于区分客户表面说法与真实任务，识别“想要一个 AI”背后的真实目标、当前替代方案和核心痛点。
- `Event Storming`：作为业务流程还原机制，用于把聊天、文档、SOP、工单等材料重新组织成“事件 - 角色 - 动作 - 系统 - 异常点”的业务链路。
- `需求基线 / 变更管理`：作为状态管理机制，用于维护已确认、待确认、冲突项、缺失项和版本演进，确保每轮澄清都在前一轮基础上收敛，而不是反复重来。

#### 7.5.2 平台内部工作流

1. `摄取`：接收自然语言、文档、表格、图片、会议纪要、群聊记录等原始输入，建立证据池。
2. `结构化`：基于 BABOK 骨架抽取业务目标、角色、流程、系统、规则、约束和风险，形成项目画像初稿。
3. `判断真实需求`：基于 JTBD 判断客户真正想完成的任务、当前替代方案和核心痛点，识别假需求和伪 AI 需求。
4. `还原业务流程`：基于 Event Storming 思路还原业务运行链条，识别触发点、处理动作、系统边界、异常点和潜在 AI 介入点。
5. `检测冲突与缺失`：识别材料之间、口头表述之间和版本之间的矛盾，提取缺少的关键前提与待确认事项。
6. `关键追问`：只生成最影响判断与收敛的 3-5 个问题，不做无限追问。
7. `当前理解回显`：输出当前已确认事实、待确认项、范围边界和需求定义草稿，供客户确认或修正。
8. `版本化确认`：在每轮确认后形成需求定义版本，并记录新增证据、状态变化和关键判断。

#### 7.5.3 平台需要持续维护的核心对象

- `原始证据`：原始聊天、文档、图片、表格和访谈内容。
- `结构化业务画像`：目标、角色、流程、系统、规则、约束、风险。
- `业务流程骨架`：按事件、动作、系统、异常点组织的流程视图。
- `需求状态清单`：已确认、待确认、冲突项、缺失项。
- `需求定义版本`：每轮确认后的问题定义、边界、MVP 建议和变更记录。

#### 7.5.4 方法论落地原则

- 方法论不是展示给用户看的章节标题，而是平台内部的抽取规则、判断规则、追问规则、状态更新规则和输出模板。
- 平台的目标不是“懂很多理论”，而是稳定完成需求确认。
- 每次新增信息都应优先更新状态和证据，而不是直接重写整份结论。
- 最终输出必须能回答三个问题：我们理解的是什么问题、哪些内容已经确认、接下来还能否继续推进。

## 9. 技术方案

### 9.1 总体架构

建议采用“多模态摄取 + 结构化上下文 + 分阶段分析 + 工作台沉淀”的架构，而不是一个大 prompt 直接跑到底。

系统分为六层：

- 前端工作台：三栏工作台与阶段切换
- 接入层：聊天、文档、图片上传
- 解析层：文档抽取、OCR、图片理解、文本切块
- 理解层：项目画像、角色、流程、目标、约束、风险建模
- 判断层：缺口识别、冲突检测、AI 适配判断、追问生成
- 产出层：MVP 方案、原型建议、结构化结论、导出

### 9.2 推荐技术栈

- 前端：Next.js + TypeScript
- 后端：FastAPI
- 异步任务：Redis + RQ
- 数据库：PostgreSQL + pgvector
- 文件存储：S3 / MinIO
- AI 能力：多模态模型 + 通用推理模型 + embedding 模型

### 9.3 关键实现原则

- 聊天记录不是唯一状态，结构化上下文才是核心状态
- 优先做增量更新，不每轮全量重算
- 方案起草必须建立在足够的证据和确认度之上
- 所有关键结论尽量保留 source refs

## 10. 核心数据模型

- Project：项目基础信息与阶段状态
- Source：原始资料与解析状态
- EvidenceChunk：可检索的证据片段
- ProjectProfile：业务目标、流程、角色、系统、约束
- RequirementItem：事实 / 假设 / 风险 / 待确认项 / 范围 / 目标
- SolutionDraft：AI fit、MVP 范围、能力包、风险、验收指标
- PrototypeSuggestion：页面、模块、交互流、原型说明

## 11. MVP 范围

### 11.1 必做

- 聊天 / 文档 / 图片输入
- 文档解析与 OCR
- 项目画像抽取
- 待确认项生成
- 真实需求收敛
- MVP 方案生成
- 原型建议生成
- 三栏工作台展示

### 11.2 暂不做

- 高保真视觉自动生成
- 自动代码生成
- 复杂权限与多人审批流
- 深行业专家系统
- 自动报价 / 合同

## 12. 成功指标

- 需求收敛周期缩短 30%+
- 需求分析结果可直接进入设计 / 研发评估
- 每个阶段都有清晰产物，而不是只留下聊天记录
- 待确认项可持续减少，项目画像完成率稳定提升

## 13. 风险与边界

- 变成陪聊机器人，缺少结构化结果
- 过早给方案，跳过业务理解
- 把所有问题都当成 AI 问题
- 资料和结论不关联，无法追溯
- 输出泛泛而谈，无法继续执行
