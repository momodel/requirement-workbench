## 用户故事（客户能力光谱 · 一句话画出市场）

3 个真实客户来源天然拼出了"客户 AI 能力光谱"的三段，正好对应我们要接的市场全段。**Demo 时讲清这 3 段是谁、长什么样、卡在哪儿，比讲"我们功能多丰富"有效得多** —— 听众会自动对号入座。

### 第 1 段 · 客户没能力、没经验 —— 杭规院一期
- 客户画像：完全不会用 AI 讨论需求，传统乙方对接模式
- 卡点：需求不明确，只能把模糊话丢给乙方猜；视觉交互稿来回反复，最后还是做错
- 深层原因：客户讲不清自己想要什么，乙方猜也猜不准，中间传话成本巨大
- 我们怎么接：客户**只需要瞎聊**，agent 主动追问把模糊话拆成可决策的颗粒；乙方/PM 看右栏就知道客户当前理解到哪、还有什么待确认

### 第 2 段 · 客户已会用通用 AI，但落不到实施层 —— 杭规院二期
- 客户画像：客户自己会用 gemini deep research，能产出概念层面的方案
- 卡点：deep research 给的是"思路 + 大纲"，但**到了实施层就断了** —— 没有可签收的需求文档、没有可点击的交互稿、没有可追溯到原始 source 的引用
- 深层原因：通用 AI 是回答工具，不是落地工具；deep research 不带项目沉淀、不带 grounded RAG、不出可交付物
- 我们怎么接：deep research 之后的"最后一公里" —— 同样的输入在我们产品里走完整链路，最后产出**对照得出来**的差异（grounded 引用 / 沉淀总集 / 可点交付物）

### 第 3 段 · 客户知道 AI 能帮，但讲不清要做什么 —— 省文投 OPC
- 客户画像：知道 AI 可以赋能 OPC，找乙方做但**自己讲不清**要做什么、也不知道 AI 的能力和边界
- 卡点：客户的话是"AI 帮我做点什么"，乙方接到的是空集；双方反复扯皮，扯到最后都没共识
- 深层原因：客户对 AI 的认知是"它能做但具体做啥不知道"，乙方对客户业务的认知是"我懂技术但不懂你业务"，**双方在不同维度上模糊，根本对不上**
- 我们怎么接：agent 带 BABOK / JTBD / Event Storming 做**双向探查** —— 一边问客户业务、一边亮自己 AI 能力的边界，让双方在同一张沉淀总集上对齐

### 一句话总结
- **一期 = 客户没能力**（agent 替客户思考）
- **二期 = 客户有能力但落不了地**（接 deep research 的最后一公里）
- **OPC = 客户有意愿但讲不清**（双向探查 + 边界对齐）

→ Demo 的客户案例页可以做成一张「客户 AI 能力光谱」横轴图，把 3 个客户点上去，然后说"我们的产品要能接整段光谱，不是只接其中一段"。

## 项目迭代故事（4 条演进线 · 一致的 meta 模式）

4 条线背后是同一个模式：**先借通用工具试水 → 撞到通用工具的能力边界 → 沉淀成专业产品形态**。这条 meta 模式本身就是 Demo 中"为什么我们不是重复造轮子"的最有说服力的回答 —— 我们不是先有产品想法再去找问题，我们是被现实一步步逼出来的产品形态。

### 演进线 1 · vibe design：从"AI 出 HTML 截图"到"gpt-image-2 + design-md 精细化"
- 起点：在 OpenClaw（龙虾）里 vibe 聊，让 AI 出 HTML 界面，截图回来或写进带界面的需求文档
- 撞墙：实际开发时发现 **AI 生成的 HTML 太糙，不能直接依据来开发**
- 演进到：开发过程改用 gpt-image-2 连续生成成套界面稿，再结合 design-md 项目精细化设计
- meta 收获：**"能看"和"能用"是两回事** —— 早期 vibe 出来的稿足够 demo，但要真落地必须升级到精细化产线

### 演进线 2 · 知识库：从 NotebookLM CLI 到自建 RAG + LLM Wiki 双层
- 起点：项目本来就类似 notebooklm，让客户丢资料；为了省时间直接用了 NotebookLM CLI
- 撞墙：**用起来是和 NotebookLM 进行对话，很慢、很不稳定**；而且角色错位 —— NotebookLM 把自己当主体，我们的 agent 反而变成它的客户
- 演进到：自建 RAG（Docling + Qdrant + LlamaIndex）→ 进一步借鉴 karpathy 的知识库思路加一路 LLM Wiki
- meta 收获：**通用工具的"角色定位"和我们要的"角色定位"经常错位** —— NotebookLM 是个独立产品，不是组件；要把它当组件用就会反复打架。自建后还能进一步分层（grounding 客观证据 vs 长期理解），是通用工具给不了的

### 演进线 3 · 产物：md → md+HTML → md+gpt-image-2+HTML
- 起点：交付物只有 markdown 文档
- 第一步演进：要做界面 / 交互示意 → 加 HTML 产物
- 第二步演进：gpt-image-2 出来后 → 用它先出精细界面图，再 → HTML，效果远好于直接生 HTML
- meta 收获：**交付物不是单一形态** —— 文档讲清"是什么"，页面方案讲清"长什么样"，交互稿讲清"怎么用"。三件套缺一不可，**让客户摸一摸比一千句需求描述都管用**

### 演进线 4 · 输入方式：文档 → 文档 + 富媒体 + 实时对话
- 起点：只支持各种格式的资料 / 文档输入
- 撞墙：**实时对话也是很重要的输入方式** —— 客户可能随时有个想法、需求点想丢进来让智能体分析
- 演进到：把对话本身也作为一类输入；下一步 → 实时音频转写、PWA 移动端，让客户在车上、电梯里、会议间隙的灵感不丢
- meta 收获：**客户的输入是异步、碎片、多模态的** —— 你不能假设客户会"坐下来一口气说清楚"。产品要接住所有形态，不挑客户

### 横向 meta（4 条线连起来）
| 线 | 起点（试水） | 撞到的墙 | 演进结果 |
|---|---|---|---|
| design | vibe HTML | AI 生稿太糙 | gpt-image-2 + design-md |
| 知识库 | NotebookLM CLI | 慢 / 不稳 / 角色错位 | 自建 RAG + LLM Wiki 双层 |
| 产物 | 纯 md | 客户看不到样子 | 文档 + 页面 + 交互三件套 |
| 输入 | 文档 | 客户想随时丢点子 | 文档 + 富媒体 + 实时对话 |

**统一发现：通用工具能让我们 demo 起步，但每一条线都会在"真用"那一刻撞墙**。撞墙之后我们的选择都是同一个 —— **沉淀成专业产品形态**，而不是接受"通用工具的将就"。这就是为什么这个产品不是"把通用智能体重新包一遍"，是把"用通用智能体撞到的所有墙"逐一翻成产品形态。

→ Demo 时这一节适合放在"我们怎么走到今天"章节，紧接着上面的客户故事。客户故事讲"接谁"，演进故事讲"我们为什么长成现在这样"，两段合起来就是一个完整的"为什么是这个产品 + 为什么是我们做"的答卷。

## Demo 故事梳理 Q&A（2026-04-27）

### 已对齐的核心信号

- **受众**：复合 — 技术评审 + 客户/甲方 + 投资人/合作方，一份 PPT 多用
- **主案例**：杭规院二期为主线（"客户已会 gemini deep research 但落不到实施层"），一期作为光谱另一端补充（说明产品覆盖"完全不会用 AI"到"会用但落不了地"的全段客户能力曲线）
- **目标用户**：客户（甲方需求方）为先；PM / 需求分析师 / 一切有需求对接动作的岗位次之
- **递归 angle**：作为 mic-drop —"把我们自己产品的需求丢进 seed project 跑一遍来验证产品自己"
- **"不抗拒地聊"**：不是单一功能，是 **专业产品定位 + 首页技术展示 + 用户案例 + UI 交互 + agent 主动追问** 共同打造的氛围合力

### A · vs 通用智能体（codex / claude code / 豆包 / ChatGPT）的真差异

通用智能体表面都能做"项目级沉淀 / 方法论 prompt / HTML canvas"，所以表面对比立不住。真差异在 **产品形态**，不是组件能力：

1. **使用门槛**：通用智能体是 toolkit，给"会用 AI 的人"用（codex 要 cd 目录、claude code 要懂命令行、ChatGPT canvas 要客户自己开/分/管）。我们是"打开就能讲"的产品壳，不是工具箱。客户的 cognitive load 是产品形态决定的，不是 prompt 能改的。

2. **对话角色翻转**：通用智能体设定是"用户主导，AI 辅助"；我们设定是"agent 主导，客户辅助"，UI 上 agent 提问占主视觉、客户输入框是配角。这种翻转要靠 UI + 流程 + 沉淀总集 + 阶段 rail 共同撑起来，**单靠 system prompt 翻不过来**。

3. **沉淀的形态**：codex 的项目沉淀是 git/files，是程序员看的；我们的 7 类沉淀是需求方看得懂的产品 UI（待确认 / 冲突 / MVP / 版本快照……）。**「程序员视角的 git」vs「需求方视角的看板」的根本差异**。

4. **整合度（专业感）**：通用智能体让客户自己拼 RAG + 主动追问 + 自动归档 + 阶段推进 + 可点交互稿，客户拼不动。我们交付的是已组装好的产品，客户感知是"专业的人帮我做了专业的事"，而不是"AI 工具我得自己 DIY"。

5. **专业产品的信任锚**：产品名「客户需求转译台」+ 首页技术栈 + 用户案例 + 方法论 skill 全明牌，客户开门第一秒就建立"这是干这事的专业产品"心智。通用智能体打开第一秒是"我接下来要 prompt 什么"。**信任感是定位 + 视觉 + 案例堆出来的，不是功能能补的**。

→ 一句话：**通用智能体是给开发者的 toolkit；我们是给需求方的 product**。组件可重叠，产品形态不可代替。

### B · vs NotebookLM

和 A 同类逻辑：**专业产品本身就让客户放心**。客户拿到一个专门为"需求转译"做的产品，第一秒就能信"专业的人在做专业的事"；NotebookLM 是通用工具，丢给客户用客户会抗拒。Demo 时不必逐项拆"我们针对这个场景做了多少优化"，定位本身就是答案。

补充技术差：NotebookLM 是 RAG 即产品（被动答疑）；我们是分析 agent 内嵌 RAG（主动追问 + 自动归档 + 出交付物）。NotebookLM 是中间产物，我们是终点。数据落地上，NotebookLM 是 Google SaaS，ToG 客户（杭规院 / 省文投）不能上传，我们本地化。

### C · 商业 / 客户视角

- 目标用户：理想是客户直接来用；同时也可以是 PM / 需求分析师 / 任何有需求对接动作的岗
- 客户为什么付费：deep research 之后的"最后一公里"——把概念落到实施层
- 计费模型 / 单价：待定（按项目 vs 订阅 vs 部署，需进一步明确）

### D · Critical 自我批判（答案）

12. **客户用了真能一个人完成需求分析吗？** 不能完全。能把"客户能独立完成的部分"从 ~0% 提到 60-70%，剩下 30-40%（取舍 / 落地决策）仍要乙方/PM 介入。我们不消灭中间层，让甲乙双方都更高效。**「工具 + 服务」混合，不是纯工具**。

13. **ChatGPT + Notion + 自己整理 vs 我们？** 差在 cognitive load。通用工具是被动的——客户得记得"这次聊完要把结论 copy 到 Notion 哪个分类"；我们是主动的，agent 自动归档。**客户不需要会用产品，只需要能讲**。三件免费工具拼出来的成本，省下来的反而是客户的脑子。

14. **哪种客户不适合？**
    - 需求已极其明确、就缺执行（用 Jira/Linear 就行）
    - 一次性极小项目（投入产出比不划算）
    - 完全不愿意打字 / 上传 / 讲话的（产品再易用也得有输入）
    - 极度保密、连私有部署都不接受的

15. **客户内部有强 PM / 强乙方咨询时，我们替代谁？** 谁都不替代，**改变工作流**。强 PM 用我们 = 多了个 24×7 助手；乙方咨询用我们 = 把方法论产品化，乙方从"卖方法"转向"卖落地服务"。

16. **长期壁垒？** 三层：
    - L1（弱）方法论 skill / prompt — 可复制
    - L2（中）项目沉淀数据 + 沉淀格式约束 — 客户迁移成本高
    - L3（强）客户工作流绑定 + 专业产品品类心智 — 真壁垒

    大模型每 6 个月跳一次，对我们是利好 — agent 越聪明，追问越准、归档越准、交付物越好。**我们是大模型能力的"产品形态化"，模型升级我们升级，不是对手**。
