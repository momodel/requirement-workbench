requirement workbench prd

客户需求转译台

这是一个面向 AI 转型前期的需求分析与方案收敛工作台。它不是陪聊式咨询 bot，而是把客户原始表达、业务资料、流程说明和专业知识，逐步转成可执行需求、MVP 方案和设计交付物。

5 个阶段 3 栏工作台 PRD + 原型
product summary

产品定位

一句话定义

把“客户说得很乱的需求”转成“产品、设计、研发、售前都能继续推进的执行稿”。

核心问题

客户表达的是想法，不是执行需求；资料分散、流程复杂、AI 边界不清，导致前期需求澄清成本高、方案反复改。

核心价值

降低前期咨询成本，形成结构化项目画像、真实需求、MVP 方案和设计交付物，让需求分析结果直接进入设计和研发阶段。

target users

目标用户

直接用户

AI 转型咨询顾问、售前顾问、产品经理、解决方案架构师、企业内部数字化负责人。

间接受益者

客户业务负责人、设计师、研发团队、项目经理。

核心产物

项目卡片、项目画像、真实需求、待确认项、MVP 草案、信息架构、页面结构和低保真原型说明。

phase model

五阶段流程

1. 需求接入

收客户原话、背景、初始资料，建立项目卡片和缺失资料清单。

输入

原始需求、会议纪要、介绍文档、产品资料

系统动作

抽取需求要点、建立项目卡片、识别缺失资料

输出

项目卡片、资料清单、初步问题定义

2. 业务理解

从资料里还原业务流程、角色、术语、系统和约束。

输入

流程文档、角色说明、系统清单、访谈补充

系统动作

流程建模、角色抽取、术语整理、系统边界识别

输出

项目画像、角色清单、现状流程图、痛点约束

3. 需求收敛

把想法、事实、假设、边界拆开，只围绕关键缺口追问。

输入

项目画像、业务流程、冲突点、用户补充

系统动作

列待确认项、识别冲突、收敛真实需求、排序优先级

输出

已确认事实、范围边界、真实需求、待确认项

4. 方案定义

判断 AI 适配度，定义 MVP、风险边界和实施前提。

输入

已收敛需求、数据条件、系统条件、风险要求

系统动作

评估 AI 适配度、划定边界、生成 MVP、定义能力拆分

输出

MVP 草案、风险边界、实施前提、验收指标

5. 设计交付

把方案转成页面结构、交互流和可交付原型。

输入

MVP 方案、能力拆分、关键字段、业务流程

系统动作

建立页面清单、信息架构、交互流和低保真方向

输出

页面结构、页面地图、交互流、原型说明

information architecture

信息结构

顶部状态栏

显示项目名、当前阶段、资料数、待确认项、AI 适配度、页面数 / 原型版本等全局状态。

左栏：资料与证据

沉淀输入信息与证据来源，如项目概况、资料列表、流程文档、术语表、页面地图。

中栏：聊天与澄清

负责接住客户原话、做关键追问、输出当前理解和阶段推进建议。

右栏：项目收敛结果

沉淀项目画像、已确认事实、范围边界、待确认项、MVP 草案、页面结构与指标卡片。

mvp scope

MVP 范围

必须有

资料上传、项目卡片、项目画像、待确认项、真实需求、MVP 草案、页面结构建议。

先不做

高保真设计自动生成、代码自动产出、复杂行业专家系统、报价与合同自动输出。

成功标准

单个项目需求收敛周期缩短 30%+，需求分析结果可直接进入设计或研发评估。
